# 钓鱼控制问题调研与可行方案（无训练版）

## 1. 问题重述（基于仓库代码）

结合 `docs/problem_formulation.md` 和 `src/gym/fishing_env.py`：

1. 问题是部分可观测控制（POMDP），动作是二值 `a in {0,1}`。
2. 目标是“先保证成功率，再压缩成功耗时”。
3. 约束是资源紧张、60Hz（每秒60次决策）、强实时低延迟。
4. 运行时未知 `difficulty / expertise / strength`，策略不能依赖这些显式输入。

因此本文件只保留不需要训练 AI 模型的路线。

## 3. 比较有希望的无训练方案

## 方案A：轻量 Bang-Bang + 滞回/前馈（首选）

核心思路：

1. 以 `fish_center - player_center` 为主误差做开关控制（按住/松开）。
2. 增加滞回区间（hysteresis），避免在临界点频繁抖动切换。
3. 加入重力前馈与简单速度补偿（即使观测不到速度，也可用一阶差分近似）。
4. 阈值采用在线自适应（根据最近窗口的追踪误差和进度变化自动微调），不依赖显式难度参数。

为什么适配：

1. 运行时复杂度 `O(1)`，最容易满足 60Hz 且延迟稳定。
2. 无训练成本，参数可直接网格搜索或手调。
3. 与当前二值动作结构天然匹配。

风险：

1. 对极端鱼轨迹可能次优。
2. 若不做自适应，固定阈值在不同隐藏参数下会退化。

## 方案B：固定小预算滚动 MPC（非学习）

核心思路：

1. 每步只前瞻很短时域（例如 12~24 个 step）。
2. 只评估少量候选动作序列（例如固定模板或束搜索 top-k）。
3. 选择第一步动作执行，下一步再重算（receding horizon）。
4. 规划时采用“参数场景集合”（多组可能 difficulty/装备组合）并做鲁棒打分（worst-case 或 CVaR）。

为什么适配：

1. 不需要训练模型，直接复用 `FishingEnv` 做前向仿真。
2. 比纯规则控制更能处理随机轨迹扰动。
3. 可以用硬预算控制最坏时延（例如每步最多仿真 N 条轨迹），且可自然处理参数未知。

风险：

1. 比方案A更吃CPU。
2. 若预算设置不当，可能无法稳定达到 60Hz。

## 方案C：离线枚举/动态规划表 + 在线查表（非训练）

核心思路：

1. 将观测空间离散化（如鱼/玩家相对位置、相对速度估计、进度段）。
2. 离线在“参数分布”上枚举动作价值并生成鲁棒策略表（policy table）。
3. 在线只做离散映射+查表，复杂度接近 `O(1)`。

为什么适配：

1. 在线速度非常快，适合极端延迟敏感场景。
2. 无神经网络训练与推理依赖，部署简单可控。
3. 可以通过离线覆盖参数组合来应对运行时参数未知。

风险：

1. 离散粒度影响性能与表大小。
2. 需要处理插值/边界态，避免查表抖动。
3. 参数覆盖不足时，未见组合下可能性能下降。

## 4. 推荐顺序（按 60Hz 与资源约束）

1. 第一优先级：方案A（Bang-Bang + 滞回/前馈）
2. 第二优先级：方案C（离线查表）
3. 第三优先级：方案B（固定预算短视距 MPC）

说明：

1. 如果目标是“最稳妥达成 60Hz”，先上 A。
2. 如果要在不训练模型前提下继续提性能，优先补 C。
3. B 作为性能上限探索，但需严格做单步耗时预算。

## 5. 最小实验协议（无训练版）

1. 统一指标：`success_rate`、`avg_success_time`、`avg_episode_time`。
2. 增加实时指标：`p95_step_latency_ms`、`max_step_latency_ms`。
3. 先在 difficulty=5 调参，再扩展 difficulty=1~9。
4. 固定种子集合，保证方案对比可复现。
5. 增加“参数未知泛化测试”：训练/调参时不暴露参数标签，仅用观测输入评估全参数组合。
